## Networking Concepts Extracted for BlkSpace

### 1. The 5-Layer Internet Model (Ch. 1)

**What the textbook says:**
- Application (Layer 5): HTTP, SMTP, DNS
- Transport (Layer 4): TCP, UDP
- Network (Layer 3): IP, routing
- Link (Layer 2): Ethernet, Wi-Fi
- Physical (Layer 1): Copper, fiber, radio

**How BlkSpace uses it:**
- **Application:** Nostr replaces HTTP for social data; Iroh replaces HTTP for media blobs
- **Transport:** TCP for reliable relay sync; UDP for local mesh discovery
- **Network:** Town-based routing instead of global IP routing
- **Link:** Wi-Fi for online; BLE mesh for offline (Phase 2+)
- **Physical:** Tier 0 laptops (design constraint)

**Documented in:** `docs/TOP_DOWN_APPROACH.md`

### 2. Packet Switching vs. Circuit Switching (Ch. 1.3)

**What the textbook says:**
- Packet switching: Resources on demand, shared capacity, efficient for bursty traffic
- Circuit switching: Dedicated resources, guaranteed performance, wasteful if unused

**How BlkSpace uses it:**
- Traditional social media = circuit-like (always-on servers, reserved capacity)
- BlkSpace = packet-like (relay sync on demand, shared town capacity)
- More efficient for bursty HBCU usage (spikes during homecoming, events)

**Documented in:** `docs/TOP_DOWN_APPROACH.md`

### 3. Delay, Loss, Throughput (Ch. 1.4)

**What the textbook says:**
- Transmission delay: T_trans = L / R
- Propagation delay: T_prop = d / s
- Queuing delay: Variable, depends on congestion
- Total delay: Sum of all components

**How BlkSpace uses it:**
- **Transmission delay** for media: 5MB photo at 1 Mbps = 40 seconds on Tier 0
  - Mitigation: Iroh chunked transfer, progressive upload, compression
- **Propagation delay** cross-town: Nashville → DC = 3.6 ms (negligible)
  - Cross-town sync can be async, not real-time
- **Queuing delay** under load: Multiple relays per town, load balancing
- **Target throughput:** 1 Mbps sustained (works on old Wi-Fi)

**Documented in:** `docs/TOP_DOWN_APPROACH.md`, `docs/architecture-blueprint.md`

### 4. TCP vs. UDP (Ch. 2, 3)

**What the textbook says:**
- TCP: Connection-oriented, reliable, ordered (3-way handshake: SYN → SYN-ACK → ACK)
- UDP: Connectionless, best-effort, no ordering

**How BlkSpace uses it:**
- **Tauri IPC** (frontend ↔ backend) = TCP-like
  - Connection-oriented: persistent IPC channel
  - Reliable: Rust error handling, structured results
  - Ordered: sequential command execution
- **Nostr relay** = TCP (WebSocket) for reliable streaming
- **Local mesh discovery** = UDP-like (Phase 2+ BLE)

**Nostr Auth Analogy:**
- Challenge request = SYN
- Challenge string = SYN-ACK
- Signed auth event = ACK
- This is exactly TCP's 3-way handshake, applied to cryptographic authentication

**Documented in:** `docs/TOP_DOWN_APPROACH.md`

### 5. DNS (Ch. 2.5)

**What the textbook says:**
- DNS is hierarchical: Root → TLD → Domain → Subdomain
- Resolution: query → recursive resolver → authoritative server
- Trust: Certificate Authorities (CAs)
- Censorship: ICANN can seize domains

**How BlkSpace replaces it:**
- **Nostr pubkeys** = self-certifying addresses
  - No hierarchy: flat, 256-bit keys
  - No resolution needed: pubkey IS the address
  - No CAs: cryptographic signatures prove identity
  - No seizure: no central authority owns the namespace

**Documented in:** `docs/TOP_DOWN_APPROACH.md`

### 6. Multiplexing & Demultiplexing (Ch. 3.2)

**What the textbook says:**
- Multiplexing: gather data from multiple sockets, add headers, pass to network
- Demultiplexing: deliver received segments to correct sockets

**How BlkSpace uses it:**
- **Tauri IPC** multiplexes 30+ commands over one channel
  - get_user, create_post, toggle_like, send_weixbucks, etc.
  - All go through a single IPC connection
  - Rust demultiplexes to appropriate database tables

**Documented in:** `docs/TOP_DOWN_APPROACH.md`

### 7. Reliable Data Transfer (Ch. 3.4)

**What the textbook says:**
- rdt evolution: 1.0 (perfect) → 2.0 (bit errors) → 2.1 (ACK/NAK) → 2.2 (NAK-free) → 3.0 (loss + timers) → TCP (pipelining, congestion)

**How BlkSpace uses it:**
- **Checksums:** CRC in SQLite, SHA-256 in blob store
- **ACK/NAK:** Nostr OK/EVENT messages
- **Timers:** React Query retry with stale-while-revalidate
- **Pipelining:** Batch event sync (sync_recent in relay_manager)
- **Congestion control:** Rate limiting (30 req/60s per pubkey)

**Documented in:** `docs/TOP_DOWN_APPROACH.md`

### 8. Error Detection (Ch. 3.3, 6.2)

**What the textbook says:**
- **Internet checksum:** Sum 16-bit words, 1s complement, check all 1s
- **CRC:** Cyclic Redundancy Check detects burst errors with high probability
- **Parity:** Single-bit parity detects odd errors; 2D parity better but still weak

**How BlkSpace uses it:**
- **Nostr event IDs:** SHA-256 hash of serialized event (detects tampering)
- **Blob integrity:** SHA-256 of file content (deduplication + integrity)
- **CRC:** Handled by hardware (Ethernet/Wi-Fi NICs) — transparent to application

**Key insight:** CRC is hardware-fast; SHA-256 is software-strong. BlkSpace uses both (CRC at link layer, SHA-256 at application layer).

**Documented in:** `docs/TOP_DOWN_APPROACH.md`

### 9. P2P Distribution (Ch. 2.6)

**What the textbook says:**
- Client-Server: D_cs = max(NF/u_s, F/d_min)
- P2P: D_p2p = max(F/u_s, F/d_min, NF/(u_s + Σu_i))
- **Key insight:** More peers in P2P = faster distribution (not slower!)

**How BlkSpace uses it:**
- **Iroh content-addressed sharing** = P2P for media blobs
- More nodes pinning content = faster retrieval
- Town relays act as both clients and servers
- Content is addressed by hash (CID), not location

**Documented in:** `docs/TOP_DOWN_APPROACH.md`, `docs/hub-theory.md`

### 10. Multiple Access Protocols (Ch. 6.3)

**What the textbook says:**
- **Random Access:** ALOHA, CSMA/CD (collision → backoff)
- **Channel Partitioning:** TDMA, FDMA (fixed allocation)
- **Taking Turns:** Token Ring (no collisions)

**How BlkSpace uses it:**
- **Nostr relay mesh** = "Taking Turns" + "Random Access" hybrid
  - Each relay has its own "turn" (town-based)
  - Within a town, events are broadcast (random access)
  - No global collisions because towns are partitioned

**Documented in:** `docs/TOP_DOWN_APPROACH.md`

## Security Papers from the Export

### 1. Kimura et al. 2025 — Nostr Attacks (EuroS&P)

**What the paper says:**
- Signature verification bypasses in Nostr clients
- Unauthenticated CBC encryption in **NIP-04** legacy DMs (AES-256-CBC without MAC)
- CBC malleability + link preview = confidentiality breach
- Cache poisoning in client implementations

**How BlkSpace mitigates:**
- **No NIP-04 implementation** — legacy DM standard is not implemented (vulnerable CBC)
- **No NIP-17 implementation** — modern sealed-box DMs also treated as untrusted per paper's general caution
- **Disable automatic link previews** (prevents CBC malleability attack)
- **Strict signature verification** on every event
- **Town tag filtering** (reject events without `t:hbcu-town:`)
- **Nostr DMs marked as untrusted** for sensitive communication

**Documented in:** `docs/security-considerations.md`

---

### 2. Fausak et al. 2024 — MIDF (AICCSA)

**What the paper says:**
- Relationship-based (graph) anomaly detection
- Malicious Intent Vector: composite score of suspicious signals
- ML-driven, extensible framework
- Detects Sybil attacks, coordinated inauthentic behavior

**How BlkSpace uses it:**
- **Engagement Quality multipliers** (0.25 for Sybil spikes, 0.1 for star patterns)
- **Rate limiting** (30 req/60s)
- **Economic anomaly detection** (suspicious reward patterns)
- Future: ML-based detection on Tier 2+ nodes

**Documented in:** `docs/reward-formulas.md`, `docs/security-considerations.md`

---

### 3. Biagioni 2026 — BitChat Protocol (ICNC)

**What the paper says:**
- Dual transport: BLE mesh (local/offline) + Nostr (global)
- TTL-limited flooding (max 7 hops)
- Noise Protocol encryption
- No accounts, no phone numbers, no central servers

**How BlkSpace uses it:**
- **Borrow philosophy, not full stack**
- BLE mesh for offline local communication (Phase 2+)
- Nostr for global reach when online
- Bridge: local events signed Nostr-style, sync to relays on reconnect

**Documented in:** `docs/TOP_DOWN_APPROACH.md`, `docs/architecture-blueprint.md`

### 4. Solana Security Analysis (from Grok)

**What the analysis says:**
- Solana's account-based model requires explicit checks
- Most common vulnerability: missing signer/ownership checks
- PDA derivation, integer overflow, unsafe CPI
- Programs are upgradable by default (authority risk)

**How BlkSpace uses it:**
- **Use Solana only for high-value settlement** (Phase 4)
- **Keep micro-economy off-chain** (WeixBucks on Nostr + SQLite)
- **All Anchor code goes through strict review**
- **Defer until economy is proven** (Phase 1-3 = simulated)

**Documented in:** `docs/solana-security.md`
